摘要
本项目面向采购文件编制场景，设计并实现一个基于大语言模型的采购文件智能生成系统。系统通过“需求语义理解 + 动态模板组装 + 合规风险预审”的流程，将用户输入的非结构化采购需求转化为结构化招标参数，并自动匹配招标书、合同条款等模板，最终生成可预览、可下载、可人工复核的采购文件。
文档同时给出完整的敏捷开发过程、系统架构、前后端模块、数据库表、接口规范、提示词模板、测试用例和部署说明，可直接作为实训项目交付文档，也可作为后续 AI 开发助手生成代码的输入说明。
说明：合规审查模块在课程实训中以“规则库 + 大模型解释 + 人工确认”的方式实现，不替代真实法律意见；涉及法规或企业制度的条款应由管理员维护并定期更新。

1 项目概述
1.1 选题来源
本项目对应学校实训命题项目列表中的第 5 项“采购文件智能生成（招标书、合同条款模板填充）”。该选题要求设计基于“需求语义理解 + 动态模板组装”的合规高效采购工具，重点掌握大模型将非结构化采购需求转化为结构化招标参数的方法，并理解动态模板引擎在合同条款自动匹配、填充与风险预审中的作用。
1.2 项目背景
传统采购文件编制通常依赖人工复制模板、手动填写参数、反复检查条款一致性。对于课程实训而言，该过程适合作为大模型落地应用场景：需求输入天然是自然语言，文档生成需要结构化数据，合规预审需要可解释结果，最终成果又需要以 Word 文档形式交付。
本系统将采购员的“自然语言需求”作为入口，将大模型用于需求解析、条款推荐和风险说明，将模板引擎用于可控生成，避免完全依赖大模型自由发挥造成格式不稳定或内容不可追溯。
1.3 项目目标
实现采购需求自然语言输入，并自动抽取项目名称、预算、采购方式、交付周期、供应商资格、评分标准等关键字段。
实现招标书模板和合同条款模板的动态组装，支持模板变量填充、条款选择、文件预览和 Word 导出。
实现基础合规风险预审，包括缺失项提示、限制性条款提示、评分项不合理提示和合同风险提示。
采用敏捷开发方式组织项目，将需求拆分为用户故事、Product Backlog、Sprint 计划和验收标准。
形成可展示、可运行、可答辩、可继续扩展的课程实训项目。
1.4 项目范围
| 范围类型 | 包含内容 | 不包含内容 |
| --- | --- | --- |
| 业务范围 | 招标书生成、合同条款模板填充、风险预审、历史记录、文档导出。 | 真实招标平台对接、真实电子签章、真实供应商投标流程。 |
| 数据范围 | 课程样例数据、管理员维护的模板与规则库、用户输入的采购需求。 | 真实企业采购数据库、涉密文件、真实法律数据库实时同步。 |
| AI 范围 | 调用 OpenAI-compatible 大模型接口或本地模拟模型，完成解析、推荐和解释。 | 模型训练、微调、海量知识库构建。 |

1.5 项目价值
效率提升：将采购文件初稿生成从人工整理改为输入需求后一键生成，减少重复复制模板。
标准化：统一模板、统一字段、统一条款库，降低不同人员写作风格差异。
合规兜底：通过规则库和大模型解释提示潜在风险，帮助用户在提交前进行人工复核。
可扩展：模板、条款、规则均可配置，后续可扩展为合同审查、供应商准入或采购知识库。

3.6 Definition of Done
功能可在本地环境启动并完成演示闭环。
接口有明确输入输出格式，异常时返回可读错误信息。
前端页面无明显报错，核心按钮和表单可用。

4.3 模块划分
| 模块 | 功能说明 | 关键输入/输出 |
| --- | --- | --- |
| 需求解析模块 | 调用大模型或 Mock 解析器，将自然语言转换为结构化参数。 | 输入：需求文本；输出：ProcurementSpec JSON。 |
| 模板管理模块 | 维护招标书模板、合同模板、变量说明和适用条件。 | 输入：模板内容；输出：模板列表与模板详情。 |
| 条款推荐模块 | 根据项目类型、金额、采购方式等匹配候选条款。 | 输入：结构化需求；输出：条款清单和推荐理由。 |
| 风险预审模块 | 执行规则校验并调用大模型生成可读解释。 | 输入：结构化需求与条款；输出：风险等级和建议。 |
| 文档生成模块 | 使用 Jinja2 与 python-docx 生成 Word 文件。 | 输入：模板、参数、条款；输出：docx 文件路径。 |
| 历史记录模块 | 保存生成任务、风险摘要、导出文件、用户修改记录。 | 输入：任务数据；输出：历史列表与详情。 |

4.4 核心业务流程
步骤 1：用户在前端输入采购需求，并选择生成类型（招标书/合同条款/两者都生成）。
步骤 2：后端接收需求文本，调用大模型解析为结构化 JSON，并返回字段置信度和缺失项。
步骤 3：系统根据采购类型和参数匹配模板，推荐适用条款。
步骤 4：风险预审模块执行规则检查，输出风险等级、原因和修改建议。
步骤 5：用户确认参数后，文档生成模块填充模板并生成 Word 文件。
步骤 6：系统保存历史记录，用户可下载文件或重新生成。
5 核心功能设计
5.1 需求输入与智能解析
需求输入页面采用步骤式表单。第一步输入自然语言需求，第二步展示系统解析出的结构化字段，第三步提示缺失项并允许用户修改，第四步确认生成。这样既体现 AI 自动化，也保留人工确认，符合采购文件场景的严谨性。
| 字段 | 说明 | 示例 |
| --- | --- | --- |
| project_name | 项目名称 | 人工智能实验室服务器采购项目 |
| purchase_type | 采购类型 | 货物 / 服务 / 工程 |
| procurement_method | 采购方式 | 公开招标 / 竞争性谈判 / 询价 |
| budget | 预算金额 | 800000 |
| delivery_period | 交付周期 | 30 天 |
| qualification | 供应商资格条件 | 近三年高校信息化项目案例、售后服务能力 |
| technical_specs | 技术参数 | GPU 显存、CPU、内存、存储、质保等 |
| evaluation_factors | 评分因素 | 价格、技术、商务、售后服务 |

5.2 模板管理
模板分为招标书模板、合同条款模板和风险规则模板。模板采用“固定章节 + 可变字段 + 条件条款”的方式管理，既保证格式稳定，也能根据不同采购场景自动调整内容。
| 模板类型 | 主要内容 | 变量示例 |
| --- | --- | --- |
| 招标书模板 | 项目概况、采购需求、投标人资格、投标文件要求、评标方法、合同草案。 | {{project_name}}、{{budget}}、{{technical_specs}} |
| 合同条款模板 | 合同标的、交付验收、付款方式、违约责任、保密条款、售后服务。 | {{delivery_period}}、{{warranty_period}}、{{payment_terms}} |
| 风险规则模板 | 缺失项规则、限制性条款规则、评分比例规则、合同风险规则。 | IF budget is empty THEN high risk |

5.3 条款推荐
条款推荐采用“规则优先 + 大模型解释”的策略。系统先根据结构化字段进行硬规则匹配，例如采购类型为“货物”时加入交付验收条款，金额较高时加入履约保证金条款；再由大模型生成推荐理由和可读说明。
| 条件 | 推荐条款 | 推荐理由 |
| --- | --- | --- |
| 采购类型 = 货物 | 交付与验收条款、质量保证条款 | 货物采购需要明确交付时间、数量、验收标准和质量责任。 |
| 包含软件/系统服务 | 知识产权条款、数据安全条款、运维服务条款 | 信息化服务可能涉及软件授权、数据处理和长期维护。 |
| 预算较高 | 履约保证金条款、付款节点条款 | 高金额项目需要降低履约风险并明确付款条件。 |
| 交付周期紧张 | 延期交付违约责任条款 | 交付周期短时需要明确延期责任和补救措施。 |

5.4 合规风险预审
合规预审不直接替代法律审查，而是作为“初筛”和“提醒”。系统将风险分为高、中、低三个等级，并以结构化形式输出。
| 风险类型 | 触发条件示例 | 提示建议 |
| --- | --- | --- |
| 缺失项风险 | 预算、交付周期、验收标准、付款方式为空。 | 提示用户补充缺失字段，否则生成文件不完整。 |
| 限制性条款风险 | 资格条件出现明显排他性表述，如指定品牌、指定唯一供应商。 | 建议改为性能指标或服务能力要求。 |
| 评分不合理风险 | 价格分过低或主观分占比过高。 | 建议明确评分维度和量化标准。 |
| 合同履约风险 | 未约定验收、质保、违约责任或付款节点。 | 建议补充关键合同条款。 |

5.5 文档生成与导出
文档生成模块的核心是将“结构化采购参数 + 推荐条款 + 风险修正结果”写入 Word 模板。系统生成的 Word 文件应包含封面、目录、采购公告、采购需求、投标人资格、评标方法、合同主要条款、附件等内容。
支持生成招标文件初稿：用于展示完整采购文件结构。
支持生成合同条款清单：用于展示关键合同条款自动填充能力。
支持导出风险报告：用于展示 AI 风险预审和可解释性。
支持历史文件下载：用于展示系统可追溯性。

8 大模型与提示词设计
8.1 大模型调用策略
系统将大模型能力封装为 LLM Adapter，前端和业务服务不直接依赖具体模型。LLM Adapter 提供 parse_requirement、recommend_clauses、explain_risks 三个核心方法。若没有真实 API Key，系统自动切换到 Mock 模式，保证课程演示不受外部服务影响。
8.2 需求解析提示词
你是采购文件生成助手。请从用户输入的采购需求中抽取结构化字段。
要求：
1. 只输出 JSON，不要输出解释性文字。
2. 字段缺失时填写 null，并在 missing_fields 中列出。
3. 不要编造用户没有提供的金额、日期、品牌或资质。
4. qualification、technical_specs、evaluation_factors 使用数组。
输出 JSON 字段：
project_name, purchase_type, procurement_method, budget, delivery_period,
qualification, technical_specs, evaluation_factors, payment_terms,
acceptance_criteria, warranty_period, missing_fields, confidence.
用户需求：{{input_text}}
8.3 条款推荐提示词
你是采购合同条款推荐助手。请根据结构化采购参数，从候选条款中选择适用条款。
要求：
1. 输出 JSON 数组。
2. 每个条款包含 clause_id、title、reason、risk_level。
3. 只能选择候选条款中的内容，不要生成不存在的条款编号。
4. 如果参数不足，请说明需要用户补充的信息。
结构化采购参数：{{parsed_spec}}
候选条款：{{candidate_clauses}}
8.4 风险解释提示词
你是采购文件合规预审助手。请根据规则命中的风险项，生成简洁、可解释的风险报告。
要求：
1. 不提供法律意见结论，只提供风险提示和修改建议。
2. 每条风险包含 level、type、reason、suggestion。
3. 建议必须具体、可执行。
4. 语气保持中性、专业。
结构化采购参数：{{parsed_spec}}
命中的规则：{{triggered_rules}}
8.5 防幻觉约束
要求模型只输出 JSON，方便后端校验和纠错。
明确禁止编造金额、日期、资质、品牌、法规条文编号。
所有最终文档内容先由模板引擎控制，大模型只负责字段抽取和解释。
引入人工确认步骤，用户确认后才生成最终文档。
规则库中的法规、制度、企业红线由管理员维护，模型不直接声明“最新法律结论”。

9.3 文档生成伪代码
def generate_procurement_doc(input_text: str, template_id: int):
# 1. 解析需求
parsed_spec = llm_adapter.parse_requirement(input_text)
# 2. 校验缺失字段
missing_fields = validate_required_fields(parsed_spec)
# 3. 推荐条款
candidate_clauses = clause_repo.list_by_type(parsed_spec.purchase_type)
selected_clauses = clause_service.recommend(parsed_spec, candidate_clauses)
# 4. 风险预审
triggered_rules = risk_rule_engine.check(parsed_spec, selected_clauses)
risk_report = llm_adapter.explain_risks(parsed_spec, triggered_rules)
# 5. 模板渲染
template = template_repo.get(template_id)
rendered_markdown = jinja2.render(template.content, parsed_spec,
selected_clauses, risk_report)
# 6. Word 导出
file_path = docx_exporter.export(rendered_markdown,
project_name=parsed_spec.project_name)
# 7. 保存记录
task_repo.save(input_text, parsed_spec, risk_report, file_path)
return file_path

10.2 功能测试用例
| 编号 | 测试项 | 输入/操作 | 预期结果 |
| --- | --- | --- | --- |

10.4 验收标准
| 验收项 | 通过标准 |
| --- | --- |
| 功能闭环 | 可以从输入需求开始，完成解析、风险提示、生成、下载全过程。 |
| AI 体现 | 至少有一个环节调用大模型或 Mock 模型，输出结构化解析或风险解释。 |
| 工程完整度 | 前后端分离，接口清晰，README 可运行。 |
| 敏捷过程 | 文档中包含用户故事、Backlog、Sprint 计划和 DoD。 |
| 演示效果 | 页面可展示，生成的文档内容结构清楚。 |
| 可扩展性 | 模板、条款、规则可配置，不写死在单个函数里。 |

10.5 风险与应对
| 风险 | 影响 | 应对方式 |
| --- | --- | --- |
| 大模型接口不可用 | 无法解析需求或生成解释。 | 提供 Mock 模式和样例返回，保证演示闭环。 |
| 生成内容不稳定 | 格式混乱或出现幻觉。 | 结构化 JSON + 模板引擎，限制模型自由生成范围。 |
| 模板过复杂 | 实现时间不够。 | 优先实现一个招标书模板和一个合同条款模板。 |
| 法规条款更新 | 合规提示可能过时。 | 合规规则仅作演示，真实使用需管理员定期维护。 |
| 团队时间有限 | 功能做不完。 | 优先完成 P0 功能，模板管理可简化为内置数据。 |

12 项目总结
本项目围绕“采购文件智能生成”主题，将大语言模型能力与传统 Web 工程结合，完成从自然语言需求输入到结构化参数抽取、动态模板组装、风险预审、Word 文件导出的完整闭环。项目既体现 AI Agent/LLM 在文档自动化场景中的价值，也保留模板引擎、规则库、人工确认等工程约束，避免大模型生成内容不可控的问题。
从课程实训角度看，项目亮点包括：一是选题贴合实际业务，采购文件生成具有明确输入和输出；二是技术路线完整，覆盖前端、后端、数据库、大模型、文档导出；三是敏捷过程清晰，能够展示需求拆解、迭代计划和验收标准；四是演示风险较低，即使没有真实模型接口，也可以通过 Mock 模式完成展示。
后续可扩展方向包括：引入企业真实模板库、接入法规知识库或 RAG 检索、增加多轮问答补全需求、支持合同审查、接入 OCR 解析历史采购文件、增加审批流与权限体系。

项目名称：采购文件智能生成系统（ProcureGen AI）
项目目标：实现“需求语义理解 + 动态模板组装 + 合规风险预审 + Word 导出”的采购文件生成工具。
技术栈要求：
1. 前端：Vue3 + Vite + Element Plus。
2. 后端：Python FastAPI + Pydantic + SQLAlchemy。
3. 数据库：SQLite，提供初始化脚本和样例数据。
4. 大模型：封装 LLM Adapter，支持 OpenAI-compatible API；没有 API Key 时自动使用 Mock 模式。
5. 模板：Jinja2 渲染 Markdown/文本模板。
6. Word 导出：使用 python-docx 生成 .docx 文件。
必须实现的核心功能：
1. 采购需求输入：用户输入自然语言采购需求。
2. 智能解析：后端把输入解析为结构化 JSON，字段包括：project_name、purchase_type、procurement_method、budget、del
ivery_period、qualification、technical_specs、evaluation_factors、payment_terms、acceptance
_criteria、warranty_period、missing_fields、confidence。
3. 缺失项提示：预算、交付周期、验收标准、付款方式等为空时提示用户补充。
4. 条款推荐：根据采购类型、预算、服务内容推荐合同条款，输出推荐理由。
5. 风险预审：输出 overall_level 和 risks，每条风险包括 level、type、message、suggestion。
6. 文档生成：根据结构化参数和推荐条款生成招标文件 Word 初稿。
7. 历史记录：保存每次生成任务，包含输入文本、解析结果、风险报告、文件路径、创建时间。
8. 下载：提供 /api/export/{task_id} 下载 Word 文件。
后端接口要求：
POST /api/parse：解析采购需求。
POST /api/risk-check：执行风险预审。
GET /api/templates：获取模板列表。
POST /api/templates：新增模板。
GET /api/clauses：获取条款列表。
POST /api/generate：生成 Word 文档。
GET /api/tasks：获取历史记录。
GET /api/tasks/{id}：获取任务详情。
GET /api/export/{id}：下载文件。
前端页面要求：
1. 工作台：首页展示项目介绍、快速创建按钮、最近记录。
2. 需求输入页：大文本框、采购类型、预算、生成按钮。
3. 解析结果页：展示 JSON 字段卡片，允许编辑缺失字段。
4. 风险报告页：展示风险等级、风险列表、修改建议。
5. 文档预览页：展示生成摘要、推荐条款、下载按钮。
6. 模板管理页：展示模板和条款列表，可新增和编辑。
